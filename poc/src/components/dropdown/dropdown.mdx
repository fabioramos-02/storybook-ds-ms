# Dropdown — uso e acessibilidade

Implementado a partir da documentação pública de
[designsystem.ms.gov.br/dropdown](https://www.designsystem.ms.gov.br/dropdown/) — sem Figma de
origem (ver [`docs/08-proximos-passos.md`](../../../../docs/08-proximos-passos.md)).

## As 6 variantes da spec oficial — cobertura real

| Variante | Status | Implementação |
|---|---|---|
| Select (lista fechada, até ~15 opções) | ✅ | `.field-select` — já existia em `input.css`, é o `<select>` nativo |
| Menu (ações administrativas) | ✅ nesta rodada | `.dropdown` abaixo, `<details>`/`<summary>` |
| Combobox (filtro + digitação) | ❌ fora de escopo | exige JS de filtro dinâmico |
| Autocomplete (sugestão via API) | ❌ fora de escopo | exige integração com backend |
| Multi-select | ❌ fora de escopo | exige JS para múltipla seleção com chips |
| Searchable select | ❌ fora de escopo | combina select + busca interna, exige JS |

As 4 variantes fora de escopo exigem **comportamento real** (filtro, requisição assíncrona,
múltiplas seleções) — não cabem no modelo CSS+HTML puro. Quando alguma dessas for necessária
num projeto consumidor, o caminho é um Web Component dedicado (mesmo critério do Menu/Carousel),
não forçar isso aqui.

## Menu de ações — por que `<details>`, não JS
Mesma lógica do Accordion: abrir/fechar por clique, fechar ao perder foco/Escape e suporte a
teclado já vêm nativos do `<details>`. Evita reimplementar um gerenciador de overlay.

```html
<details class="dropdown">
  <summary>Ações</summary>
  <div class="dropdown-menu" role="menu">
    <a href="#" role="menuitem">Visualizar</a>
    <button type="button" role="menuitem">Editar</button>
  </div>
</details>
```

## Acessibilidade
- `role="menu"`/`role="menuitem"` no menu de ações (a spec oficial pede `aria-expanded`,
  `aria-controls` — no `<details>` nativo o estado já é exposto ao leitor de tela sem precisar
  replicar manualmente).
- `.field-select` herda toda a acessibilidade nativa do `<select>` (navegação por teclado,
  anúncio de opção selecionada).
- **Evitar**: dropdown que não responde a teclado — risco eliminado por usar elementos nativos
  (`<select>`, `<details>`) em vez de `<div>` com JS customizado.
