# Carousel — Web Component, uso e acessibilidade

Implementado a partir da documentação pública de
[designsystem.ms.gov.br/carousel](https://www.designsystem.ms.gov.br/carousel/) — sem Figma de
origem (ver [`docs/08-proximos-passos.md`](../../../../docs/08-proximos-passos.md)).

## Por que Web Component
A spec exige autoplay **pausável ao interagir**, controles prev/next, indicadores e navegação
por teclado — comportamento real demais para CSS+HTML puro (mesmo critério de
`<ms-header>`/`<ms-menu>`). `<ms-carousel>` centraliza isso:

```html
<script type="module" src=".../ds-sis.js"></script>
<ms-carousel autoplay interval="4000"></ms-carousel>
```

## Boas práticas aplicadas (da spec oficial)
- Recomendação de 3–5 slides (exemplo usa 3).
- Autoplay pausa em `mouseenter`/retoma em `mouseleave` (`paused` state).
- Intervalo mínimo de ~2s — default aqui é 4000ms, configurável via atributo `interval`.
- Um CTA por slide (não implementado no exemplo — slot reservado em `s.body`, próximo
  incremento se necessário).

## Acessibilidade (eMAG + WCAG 2.1 AA)
- `role="region"` + `aria-roledescription="carousel"` + `aria-label="Destaques"` no container.
- Cada slide é `role="group"` + `aria-roledescription="slide"` + `aria-label="N de M"`; slides
  fora de tela levam `aria-hidden="true"`.
- Botões prev/next/dots com `aria-label` descritivo (nunca ícone sozinho sem texto acessível);
  dot ativo marcado com `aria-current="true"`.
- Navegação por teclado: `ArrowLeft`/`ArrowRight` no container (`tabindex="0"` + `@keydown`).
- Decisão de design: **não** implementa `aria-live` no anúncio de troca de slide — avaliar se é
  necessário quando o componente for usado em produção com autoplay ativo por padrão (hoje
  `autoplay` é opt-in via atributo, reduzindo risco de leitura forçada inesperada).
